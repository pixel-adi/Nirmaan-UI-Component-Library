# Multi-Brand Theming

> [!TIP]
> **What is the Brand Axis?**
> Nirman supports four strict adopter classes (Constitution §V.2–V.5), each granting a graduated level of visual customization to match specific jurisdictional branding.

## The Three Axes

Every layout deployed via the Nirmaan architecture runs on independent visual axes calculating in real-time:

- **Brand**: `nirman` | `central-govt` | `kerala` | `tamilnadu` | `up` | `custom`
- **Mode**: `light` | `dark` | `high-contrast`
- **Density**: `default` | `compact` | `spacious`

> [!IMPORTANT]
> **Calculation Guarantee**
> Any combination of Brand × Mode × Density **MUST** inherently compile into a valid, rigorously accessible WCAG 2.1 AA UI out of the box.

## How Brand Theming Works

### In Code

Each brand is a CSS file that overrides semantic tokens:

```tsx
// Import base tokens (always required)
import '@nirman/tokens/css/primitives';
import '@nirman/tokens/css/light';

// Import a brand theme (optional — overrides primary/accent colors)
import '@nirman/tokens/css/themes/state-kerala';

// Wrap your app
<NirmanProvider defaultBrand="kerala" defaultTheme="light">
  <App />
</NirmanProvider>
```

The brand CSS uses an attribute selector `[data-brand="kl"]` which the NirmanProvider sets on the root element. This means brand + mode compose naturally — Kerala dark mode just works.

### In Figma

On Figma Professional plan, brand themes are separate library files. Designers switch brand by swapping their library subscription. On Enterprise plan, you can use Extended Collections to add brand as a mode column.

## Creating a New Brand Theme

1. Copy `packages/tokens/themes/central-govt.json` as a template
2. Override only the tokens your class permits (see table below)
3. Run `pnpm tokens:build` — Style Dictionary generates the CSS
4. Import the generated CSS in your app

### Override Scope by Class

| Token Group | Class A (Central) | Class B (State) | Class C (PSU) | Class D (Open) |
|---|---|---|---|---|
| `color.primary.*` | ✓ limited | ✓ | ✓ | ✓ |
| `color.accent.*` | ✓ | ✓ | ✓ | ✓ |
| `color.secondary.*` | ✓ | ✓ | ✓ | ✓ |
| `color.surface.*` | ✗ | ✓ tint only | ✓ | ✓ |
| `color.text.*` | ✗ | ✗ | ✓ | ✓ |
| `radius.*` | ✗ | ✗ | ✓ | ✓ |
| Typography (Latin) | ✗ | ✗ | ✓ verified | ✓ |
| Typography (Indic) | ✗ | ✗ | ✓ verified | ✓ verified |
| High-contrast mode | **never** | **never** | **never** | **never** |

**High-contrast mode is universal and never brand-customized.** This is a legal accessibility baseline.

## Theme File Configuration

Building brand packages involves defining token configurations. Because Nirman abstracts out hard-coded values from components, overriding the tokens dynamically adjusts everything across the entire workspace automatically:

```json
{
  "$class": "B",
  "meta": {
    "name": "Kerala",
    "code": "KL",
    "primary-language": "ml"
  },
  "light": {
    "color": {
      "primary": {
        "default": { "$value": "#006B3F" },
        "hover":   { "$value": "#005A35" }
      }
    }
  },
  "dark": {
    "color": {
      "primary": {
        "default": { "$value": "#4ADE80" },
        "hover":   { "$value": "#6AE89A" }
      }
    }
  }
}
```

> [!NOTE]
> Each theme file generates out to less than **1KB**. Switching environments or adding a new jurisdictional brand is purely a `configuration` update—you never need to modify UI component `.tsx` files directly!
